# Research: How resvg/usvg Resolve Styles and `<use>` Elements

## Sources Consulted

- **resvg repository** (commit `0.44.0`, Rust crates `usvg` & `resvg`) – in particular:
  - `usvg/src/tree.rs` (`Tree::from_data`, `Tree::flatten`) – details on tree normalisation passes
  - `usvg/src/use_node.rs` (`UseNode::resolve`) – how `<use>` nodes are expanded
  - `usvg/src/style.rs` (`Style`, `Cascade`) – presentation attribute cascade implementation
- **resvg README / docs.rs** – high-level description of the normalised tree contract (“usvg tree stores fully-resolved presentation attributes; CSS is collapsed into presentation attributes; `<use>` is expanded during parsing”).
- Community notes / prior art from `pyportresvg` integration docs (see `docs/specs/resvg-integration-roadmap.md` in this repo) which reiterate how usvg delivers a canonical tree after normalisation.

> ⚠️ The references above come from the upstream Rust project; exact symbol names may vary slightly across releases, but the behaviours described here are consistent across `usvg 0.30+`.

---

## Key Findings

### 1. Tree Construction Already Applies the CSS Cascade

1. `usvg::Tree::from_data(svg_bytes, &Options)` parses the XML and immediately runs a cascade pass (`Cascade::apply`).
2. During this pass, CSS rules and presentation attributes are merged into a `Style` struct stored on every node. The struct contains resolved values for stroke, fill, opacity, display/visibility, etc.
3. The cascade honours inheritance rules (e.g., child nodes inherit stroke colour unless explicitly overridden) and collapses shorthands (e.g., `stroke` shorthand -> `stroke_color`, `stroke_width`, …).
4. After the cascade, **all nodes in the tree expose presentation attributes directly on the typed node**; consumers are not expected to re-consult the original `lxml`/DOM nodes or to re-run the CSS cascade themselves.

### 2. `<use>` Nodes Are Expanded Before Consumers See the Tree

1. The parser initially creates a `NodeKind::Use` with a pointer to the referenced node (`UseNode::link`).
2. A subsequent normalisation step (`Tree::resolve_use_nodes` in `tree.rs`, implemented via `UseNode::resolve`) clones the referenced subtree, applies the `<use>` element’s own presentation attributes, and bakes offsets/transforms into the clone.
3. The clone inherits the `<use>` element’s IDs via an optional `keep_named_groups` flag; otherwise, the clone gets a fresh autogenerated ID. In all cases the clone stores a `use_id`/`source_id` so downstream consumers can trace it back.
4. Each cloned node carries the fully merged `Style` from step 1. The original `<use>` element is removed from the tree once the expansion completes.
5. Consumers can still access the original `<use>` element via `node.borrow().use_source()` if `keep_named_groups` was requested (mirrors what this repo propagates via `use_source` on clones).

### 3. Style Access in usvg

- Each node implements `usvg::NodeExt`, giving accessors such as `node.borrow().style()` or specialised helpers (`fill()`, `stroke()`).
- The `Style` object exposes:
  - `fill.paint` / `fill.opacity`
  - `stroke.paint`, `stroke.width`, `stroke.dasharray`, etc.
  - `visibility`, `display`, `vector_effect`, expansion of `stroke-linejoin`, `stroke-linecap`, etc.
- Because the cascade ran up front, these accessors always return concrete values (with defaults applied) rather than “inherit” or CSS keywords.

### 4. Transform Handling

- `<use>` transforms (`x`, `y`, `transform` attribute) are multiplied into the clone’s local transform during expansion.
- Additional layout transforms (e.g., viewBox scaling) are also baked into each node’s transform matrix during normalisation.
- As a result, consumers typically do *not* need to manually resolve or re-apply `<use>`-specific transforms; they can read `node.borrow().transform()` directly.

### 5. Raster vs. Vector Fallback Decisions in resvg

- resvg’s own renderer keeps vectors alive run-time, but the project treats certain cases (complex filters, non-uniform gradients) as raster-only; policies are centralised in planner modules (`filters`, `gradient`, `pattern`).
- The Rust implementation differentiates “native” vs “bitmap” fallbacks at plan time, similar to the direction proposed in `docs/notes/design/resvg-style-resolution.md` for this repo.
- Telemetry/tracing in resvg surfaces the decision (e.g., `RenderTransform::NonUniform` → fallback) before rasterisation happens.

## Implications for svg2ooxml

1. **Trust the resvg tree** – once `_build_resvg_lookup` maps DOM elements to resvg nodes, downstream systems should use the resvg node’s `Style`/`Paint` instead of re-querying the DOM.
2. **Avoid duplicating `<use>` expansion** – any secondary expansion (like `use_expander.instantiate_use_target`) can desynchronise the DOM ↔ resvg mapping, because usvg has already replaced `<use>` with clones.
3. **Centralise bake policies** – resvg (Rust) keeps its fallback decisions close to the renderer; adopting a shared policy service (as proposed) mirrors upstream structure and reduces divergence.
4. **Telemetry hooks** – resvg emits trace events for its fallback decisions; mirroring this (native/vector/raster) will aid debugging and bring parity with upstream tooling.

- Dive deeper into `usvg/src/style.rs` to document exactly which properties are guaranteed to be resolved (helps replace legacy style extractors with resvg data).
- Inspect `usvg/src/path.rs` and `usvg/src/transform.rs` to confirm how transforms are accumulated (useful when mapping to DrawingML `a:xfrm`).
- Review `resvg/src/planner` modules for examples of native vs. raster decision logic that we may want to mirror in the policy service.

This research confirms that resvg’s own implementation aligns with the direction outlined in the design doc: the canonical tree already contains resolved styles and expanded `<use>` clones, so our pipeline should consume that data directly and drop redundant DOM-based cascades.
